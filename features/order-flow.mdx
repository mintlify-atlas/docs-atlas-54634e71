---
title: "Order flow"
description: "How jefe and administrador roles create orders and how operarios execute them in Bodega de Frío."
---

Orders coordinate movement between zones. A supervisor creates an order specifying what to move and where; the operario sees it in their queue and executes it.

## Order types

<CardGroup cols={3}>
  <Card title="a_bodega" icon="arrow-right">
    Move a box from **Ingresos** (or **Salida**) into a specific cold storage slot in **Bodega**.
  </Card>
  <Card title="a_salida" icon="arrow-right">
    Move a box from **Bodega** to the next available position in **Salida**.
  </Card>
  <Card title="revisar" icon="magnifying-glass">
    Inspect a box in **Bodega** without moving it. The operario acknowledges the review and the order is cleared.
  </Card>
</CardGroup>

## Order fields

```typescript
type BodegaOrder = {
  id: string;              // UUID generated by createOrderId()
  type: OrderType;         // "a_bodega" | "a_salida" | "revisar"
  sourceZone: OrderSource; // "ingresos" | "bodega" | "salida"
  sourcePosition: number;
  targetPosition?: number; // required for a_bodega and a_salida
  createdAt: string;       // locale string (es-CO)
  createdAtMs: number;     // Unix ms for overdue calculation
  createdBy: Role;         // role of the user who created the order
  client?: string;
  autoId?: string;         // autoId of the source box
  boxName?: string;        // name of the source box
};
```

## Who creates orders

Only the **jefe** role can create orders. The role check is enforced in `handleCreateOrder` and `handleCreateReviewOrder`:

```typescript
if (role !== "jefe") {
  setMessage("Solo el jefe crea ordenes.");
  return;
}
```

<Note>
  The **jefe** dashboard defaults to the **Orden de trabajo** tab immediately after login. The jefe does not see the slot grid or ingresos form — only the order creation panel.
</Note>

## Creating an order

<Steps>
  <Step title="Select the order type">
    In the **Orden de trabajo** panel, choose one of the three order types: **a_bodega**, **a_salida**, or **revisar**.
  </Step>
  <Step title="Select the source box">
    Choose the source zone and position. Only boxes that do not already have a pending order are listed — this prevents double-assigning the same box (see [Pending source keys](#pending-source-keys)).
  </Step>
  <Step title="Select the target position (a_bodega only)">
    For `a_bodega` orders, select a free slot in **Bodega**. Slots that are already the target of another pending `a_bodega` order are excluded from the list.
  </Step>
  <Step title="Submit">
    The order is prepended to the `orders` array and written to Firestore. The operario sees it immediately in their queue.
  </Step>
</Steps>

## How operarios complete orders

The **operario** role sees the **Solicitudes pendientes** tab, which renders the `RequestsQueue` component.

<Steps>
  <Step title="View the queue">
    All pending orders are displayed. Each card shows the order type, source zone and position, target position, the creating role, and the creation time.
  </Step>
  <Step title="Execute the order">
    Clicking the execute button calls `executeOrder(orderId)`, which:
    - For `a_bodega`: moves the box from the source zone into the target slot, then clears the source.
    - For `a_salida`: moves the box from **Bodega** (or **Ingresos**) into the next available **Salida** position.
    - For `revisar`: confirms the box is still in its position and removes the order.
  </Step>
  <Step title="Report a failure (optional)">
    If the operario cannot complete the order, they can click the report button. This calls `handleReportOrder`, which creates an `alerta-fallo-*` alert visible to the jefe.
  </Step>
</Steps>

## Pending source keys

To prevent two orders from referencing the same box simultaneously, the dashboard maintains a `pendingSourceKeys` set derived from the current orders array:

```typescript
const pendingSourceKeys = useMemo(() => {
  const keys = new Set<string>();
  orders.forEach(order => {
    keys.add(`${order.sourceZone}:${order.sourcePosition}`);
  });
  return keys;
}, [orders]);
```

Boxes whose key is in this set are filtered out of the source selection dropdowns for new orders. The same mechanism applies to `a_bodega` target positions — any slot already reserved as a target is excluded from the free-slot list.

## Overdue orders

Orders pending for more than **2 minutes** automatically generate an alert. The dashboard runs a 30-second clock (`alertClock`) that re-evaluates `overdueOrders` continuously:

```typescript
const ALERT_DELAY_MS = 2 * 60 * 1000; // 2 minutes

const overdueOrders = useMemo(
  () => orders.filter(order => alertClock - order.createdAtMs >= ALERT_DELAY_MS),
  [alertClock, orders]
);
```

Each overdue order produces an alert with the ID prefix `alerta-orden-`. See [Temperature alerts](/features/temperature-alerts) for how these alerts are managed.

<Warning>
  Overdue alerts are only cleared when the underlying order is executed or removed. Reprogramming the order (resetting `createdAtMs` to `Date.now()`) restarts the 2-minute countdown.
</Warning>
